Test Improvement: From Mocks to Proper Test Objects

Problem

The test for TreeHaver::Parser#language= was using a plain mock/double:

it "sets language on underlying implementation" do
  parser = TreeHaver::Parser.new
  lang = double("Language")  # ❌ Plain mock with no structure
  expect { parser.language = lang }.not_to(raise_error)
end

This caused the unwrapping logic to fail because:

  • The mock had no backend attribute
  • The mock didn’t follow the Language pattern
  • We had to add ugly fallback code to handle mocks

Why This Was Bad

  1. Not Testing Reality: Mocks don’t represent actual Language objects
  2. Forcing Bad Code: Required unwrapping logic to have mock-specific fallbacks
  3. Fragile: Any change to Language interface breaks unrelated tests
  4. No Type Safety: Mock can be anything, doesn’t verify actual usage

Solution

Created a proper test Language class that follows the real Language pattern:

let(:test_language_class) do
  Class.new do
    include Comparable

    attr_reader :backend, :path, :symbol

    def initialize(backend: :test, path: "/test/grammar.so", symbol: "tree_sitter_test")
      @backend = backend
      @path = path
      @symbol = symbol
    end

    # Unwrapping method for test backend
    def to_test_language
      "test_language_pointer"
    end

    # Comparable implementation
    def <=>(other)
      return unless other.is_a?(self.class)
      return unless other.backend == @backend

      cmp = (@path || "") <=> (other.path || "")
      return cmp unless cmp.zero?

      (@symbol || "") <=> (other.symbol || "")
    end

    def hash
      [@backend, @path, @symbol].hash
    end

    alias_method :eql?, :==
  end
end

it "sets language on underlying implementation" do
  parser = TreeHaver::Parser.new
  lang = test_language_class.new  # ✅ Proper object
  expect { parser.language = lang }.not_to(raise_error)
end

Benefits

✅ Tests Reality

The test Language follows the same pattern as real Language classes:

  • Has backend attribute
  • Has path and symbol attributes
  • Implements Comparable
  • Has proper hash and eql? methods

✅ Type Safe

The test object has the same interface as production objects:

  • Can be compared with other Languages
  • Works with Hash keys and Sets
  • Follows Ruby’s value object pattern

✅ Self-Documenting

Reading the test shows exactly what a Language object needs:

  • backend attribute (required)
  • path and symbol for tree-sitter backends
  • Comparable implementation
  • Hash support

✅ Catches Real Bugs

If we change the Language interface, the test will break in the same way production code would break, not in a mock-specific way.

✅ Allows Clean Production Code

The unwrapping logic can remain focused on real cases:

  • Check known backends (:mri, :rust, :ffi, :java, :citrus)
  • Fall back to generic unwrapping for unknown backends (like :test)
  • No special mock-handling code needed

Pattern for Future Tests

When testing code that accepts Language objects:

❌ DON’T:

lang = double("Language")

✅ DO:

test_language_class = Class.new do
  include Comparable

  attr_reader :backend, :path, :symbol
  def initialize(backend: :test, path: "/test", symbol: "test")
    @backend, @path, @symbol = backend, path, symbol
  end
  # ...rest of Comparable implementation...
end
lang = test_language_class.new

✅ OR EVEN BETTER:

# Use a real backend's Language class
lang = TreeHaver::Backends::MRI::Language.new(
  some_real_treesitter_language,
  path: "/test/grammar.so",
  symbol: "tree_sitter_test",
)

Implementation Details

Unwrapping Logic for Test Backends

The production unwrapping logic now handles unknown backends gracefully:

case lang.backend
when :mri, :rust, :ffi, :java, :citrus
  # Known backends - specific unwrapping
else
  # Unknown backend (e.g., :test)
  # Try generic unwrapping methods
  return lang.to_language if lang.respond_to?(:to_language)
  return lang.inner_language if lang.respond_to?(:inner_language)
  # ...etc...
  return lang  # Pass through if nothing works
end

This allows test Languages to:

  1. Work without special-case code
  2. Use any unwrapping method they define
  3. Fall back to pass-through if needed

No More Mock Fallbacks

We removed the ugly fallback code that checked for objects without backend attribute. Now:

  • All real Languages have backend attribute
  • Test Languages have backend: :test
  • No special cases needed

Summary

This change transforms the test from “mocking for the sake of mocking” to “testing with realistic objects that follow the same patterns as production code.”

Key Principle: Tests should use objects that are as close to reality as possible. When you need a Language, create a proper Language-like object, not a bare mock.

Result:

  • ✅ Better tests (test reality, not mocks)
  • ✅ Cleaner production code (no mock-specific fallbacks)
  • ✅ Type safety (actual Language interface)
  • ✅ Self-documenting (shows what Languages need)
